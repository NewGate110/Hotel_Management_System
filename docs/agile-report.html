1. Methodology Adopted

The Grand Plaza HMS was developed using an iterative, sprint-based approach inspired by Scrum. The project was divided into three time-boxed sprints of approximately one to two days each, with a short backlog-refinement phase before development began. Because this was a solo project, formal ceremonies such as daily stand-ups were replaced by end-of-sprint self-retrospectives captured as commit messages and updated documentation. Kanban-style task tracking was used informally within each sprint to visualise work-in-progress limits and avoid multitasking across architectural layers.

Test-Driven Development (TDD) was applied throughout the backend layer: failing unit tests were written first for all business-rule classes (CancellationPolicy, PricingPolicy, PasswordPolicy) and service methods before any production code was committed. This ensured the regression safety net was in place before any refactoring or feature addition.

3. Backlog and Prioritisation

User stories were written at the start of Sprint 0 and mapped to the four roles defined in the assessment brief. Stories were sized using a simple three-point scale (small / medium / large) and assigned to sprints by value-to-effort ratio. High-value, low-risk stories such as authentication and room search shipped in Sprint 0 and Sprint 1, while lower-priority polish items (image heroes, delegated media permissions) were deferred to Sprint 2 and kept out of the critical path.

The backlog was maintained as a plain markdown plan file (docs/superpowers/plans/). Each item carried a clear definition of done: a passing test, a working UI route, and a merged commit. This prevented scope creep — if a story could not be verified end-to-end by the sprint close, it was pushed to the next sprint rather than shipped incomplete.

4. Iterative Feedback and Adaptation

Several requirements evolved during development, demonstrating the value of iterative delivery:

  • Vocabulary rebrand (Sprint 1 → Sprint 2): The initial domain model used "resort" terminology. A mid-project review identified a mismatch with the hotel management context; a targeted refactoring commit (refactor: rebrand resort theme to hotel) corrected all enum values, seeder text, and UI labels with zero database migration needed because only the C# identifiers changed, not the underlying integer values.
  • Delegated media permission (Sprint 2): An original requirement for Admin-only image management was extended after review to allow specific FrontDeskStaff members to edit room images without granting full admin access. The CanManageMedia flag was added to the database, propagated through the JWT as a conditional claim, and surfaced in the Angular auth service as a computed signal — all in a single coherent feature branch.
  • Alignment audit (Sprint 3): A systematic visual audit against the Azure Shores design system identified 26 layout defects (broken card heights, unresponsive tables, z-index stacking bugs). These were triaged by severity and resolved in order without touching business logic, showing clean separation between the UI layer and the application core.

5. Definition of Done

A story was considered done when all of the following conditions were met:

  • All backend business logic covered by at least one unit test (TDD — test written first).
  • API endpoint reachable via Swagger and returning expected HTTP status codes.
  • Angular route loads without console errors in the Docker-built bundle.
  • No hardcoded colours; all styles reference Azure Shores design tokens.

6. Retrospective Summary

The iterative approach proved effective for a full-stack solo project. Keeping sprints short (one to two days) forced prioritisation and prevented the accumulation of integration debt. The main retrospective learning across sprints was the importance of verifying compiled Angular output in the Docker bundle rather than relying on the development server alone: several CSS properties that appeared correct locally compiled to different attribute forms in the production bundle, and bundle-level grep verification became a standard part of the Sprint 3 definition of done.

If the project were repeated, Sprint 0 would include a visual design token audit before any component work began, reducing the 26-item alignment backlog that accumulated by Sprint 3.
