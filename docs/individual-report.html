1. Personal Contribution

I designed and implemented the Grand Plaza HMS in full, taking sole ownership of every layer of the stack. The system spans a .NET 10 / ASP.NET Core REST API structured in four Clean Architecture layers (Domain, Application, Infrastructure, API), an Angular 21 single-page application using standalone components and signal-based reactivity, a PostgreSQL 18.3 database managed by EF Core migrations, and a Docker Compose environment that starts the complete system with a single command.

On the backend I designed the domain model from scratch — thirteen EF Core entities covering hotels, rooms, bookings, payments, invoices, ancillary services, and a multi-role user hierarchy — and wrote the business logic for pricing, cancellation fees, and password policy as standalone, testable policy classes. Ninety-nine xUnit unit tests cover all three policy classes, the authentication service, and the booking management service; repositories are mocked throughout so tests run without a database.

On the frontend I built the complete Angular application: public pages (landing, hotel listing, hotel detail, room search, room detail), a four-role authenticated dashboard (Guest, FrontDeskStaff, HotelManager, Admin), an embedded multi-step booking wizard, role-based lazy-loaded feature modules, and a bespoke design system — Azure Shores — defined entirely through CSS custom properties. I also designed and seeded seven hotels and thirty-seven rooms with realistic images sourced from Unsplash.

2. Technical Challenges and How I Resolved Them

Clean Architecture boundary enforcement

The Clean Architecture pattern requires that the Domain layer has no outward dependencies, yet EF Core navigation properties tend to pull infrastructure concerns inward. I resolved this by defining all repository interfaces in the Domain layer and all EF Core DbSet configurations in HmsDbContext within Infrastructure, with AutoMapper profiles in the Application layer translating between domain entities and DTOs. This kept the domain completely framework-free and enabled the mock-based unit testing strategy.

JWT in an HttpOnly cookie alongside a JS-readable session

A requirement for security was to store the JWT in an HttpOnly cookie (inaccessible to JavaScript) while still letting Angular know the user's role and permissions. The solution was a dual-channel approach: the backend sets the cookie and returns a JSON LoginResponseDto body containing the session properties (role, userId, fullName, canManageMedia). Angular stores this body in sessionStorage via TokenService and exposes it as Angular computed signals. This preserved the XSS protection of HttpOnly cookies while giving the frontend the data it needs for route guards and conditional rendering.

Delegated media permissions without role escalation

The requirement to let specific FrontDeskStaff edit room images without granting them full admin rights could not be satisfied by the existing role enum alone. I introduced a CanManageMedia boolean on StaffUser, included it as a conditional JWT claim (present only when true), and added an Admin-only PATCH endpoint to toggle it per staff member. The Angular auth service reads the claim from the login response and exposes it as a canManageMedia() signal. This avoided role proliferation and followed the principle of least privilege.

EF Core model snapshot consistency

Adding the CanManageMedia column surfaced an EF Core edge case: the migration correctly set DEFAULT FALSE in SQL but the model snapshot did not include the HasDefaultValue(false) fluent API call, causing a PendingModelChangesWarning on startup. The fix required adding an explicit HasDefaultValue(false) call in OnModelCreating to match what the snapshot expected. This taught me the importance of keeping fluent API configuration and migration snapshots in lock-step.

3. Reflection on Learning

Building this system consolidated and extended my understanding in three main areas.

Object-oriented design at scale. The project put SOLID principles into practice beyond the theoretical. The Dependency Inversion Principle was enforced architecturally: high-level modules (Application services) depend only on interfaces defined in the Domain; all concrete implementations (EF Core repositories, JWT generator, BCrypt hasher) live in Infrastructure. This meant I could write 99 unit tests that mock every external dependency without ever touching a database, a file system, or the clock.

Angular signals and reactive patterns. Angular 21's signal-based reactivity model — signal(), computed(), toSignal() — is a significant departure from the RxJS-only patterns I had used previously. Learning to derive UI state from signals rather than BehaviourSubjects made the component templates simpler and change detection substantially faster, but required rethinking how to propagate state across components (answer: shared services holding signals, not event emitters).

CSS architecture for design systems. Building Azure Shores taught me that a design system is not just a palette — it is a contract. Every component must reference only the semantic tokens (--fg, --surface, --brand) rather than raw colour values. When I audited the codebase late in the project and found Tailwind zinc-* classes mixed with token-based styles, the inconsistency was immediately visible as a maintenance risk. I replaced all hardcoded colours with tokens in a single focused commit, which was only possible because the token contract was well-defined from the start.

4. Emerging Technologies

Two technology choices in this project represent relatively recent advances worth reflecting on.

.NET 10 and C# 13. .NET 10 (preview at time of writing) ships with primary constructors for all class types, collection expressions, and faster Span-based string operations. I used primary constructors throughout the Clean Architecture layers to reduce boilerplate: service constructors declare their dependencies inline rather than as separate field assignments. The performance improvements in the ASP.NET Core request pipeline (reduced allocations, faster JSON serialisation via System.Text.Json source generators) are particularly relevant to a high-throughput booking API.

Angular 21 signals. Signals represent a fundamental shift in Angular's reactivity model away from Zone.js-based change detection toward a pull-based fine-grained reactivity graph similar to Solid.js. This enables ChangeDetectionStrategy.OnPush components with true signal-driven updates, removing the need to mark views dirty manually. The interop layer (toSignal, takeUntilDestroyed) allows gradual migration from RxJS without rewriting existing services, which I leveraged for services that still use HTTP observables internally but expose their state as signals at the component boundary.

5. Conclusion

Building the Grand Plaza HMS end-to-end as a solo project required sustained application of advanced software engineering principles: Clean Architecture, TDD, iterative delivery, and a coherent design system. The most valuable lesson was that architectural discipline — enforced through dependency direction, interface-first design, and token-based styling — pays off in direct proportion to the complexity of the system. Every time a requirement changed (the vocabulary rebrand, the delegated media permission, the alignment audit), the change was localised to a single layer with no ripple effects, which would not have been possible without that foundation.
